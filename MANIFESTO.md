# Contextix Manifesto

## Software Should Remember More Than Code

Modern software development has changed dramatically.

Artificial Intelligence can now generate code, explain APIs, write tests, review pull requests, and accelerate development.

Yet one problem remains largely unsolved.

Projects forget.

Not because source code disappears.

But because understanding disappears.

---

## The Missing Layer

Today's software remembers many things.

Git remembers commits.

Repositories remember source code.

Issue trackers remember discussions.

Documentation explains pieces of the system.

But none of them truly preserve understanding.

Why was Redis chosen?

Why does this constraint exist?

Why was this architecture rejected?

Why does this module exist at all?

Developers answer these questions every day.

Then forget them.

The cycle repeats.

---

## AI Changed the Problem

Large Language Models introduced a new workflow.

Developers increasingly collaborate with AI.

Yet every conversation starts with the same burden.

Explain the project.

Explain the architecture.

Explain the constraints.

Explain previous decisions.

Eventually context fills.

The conversation resets.

The project forgets itself once again.

The bottleneck is no longer code generation.

The bottleneck is project understanding.

---

## Our Belief

Source code should not be the only memory of a project.

Projects deserve structured knowledge.

Projects deserve preserved reasoning.

Projects deserve portable understanding.

Knowledge should survive beyond conversations.

Beyond IDEs.

Beyond AI models.

Beyond individual developers.

---

## Our Vision

Contextix is not trying to replace developers.

It is not trying to replace documentation.

It is not another autonomous coding agent.

Contextix exists to preserve software understanding.

To make project knowledge portable.

To make collaboration more reliable.

To reduce repeated explanations.

To help both humans and AI understand software more deeply.

---

## Simplicity First

We believe:

Simple systems survive.

Simple architectures evolve.

Simple tools are adopted.

Every feature should justify its complexity.

If understanding does not improve,

the feature probably does not belong.

---

## Open Knowledge

Project understanding should never belong to one AI provider.

Or one programming language.

Or one company.

Project knowledge should remain portable.

Open.

Transparent.

Reusable.

This belief led to the creation of the Context Exchange Specification (CES).

---

## Engineering Before Hype

Technology changes quickly.

Engineering principles do not.

Contextix values:

Understanding over automation.

Reasoning over prediction.

Architecture over trends.

Long-term thinking over short-term excitement.

---

## Looking Forward

Perhaps one day every software project will contain more than source code.

Perhaps every repository will preserve its own understanding.

Perhaps developers will inherit not only code,

but the reasoning behind it.

If that future exists,

Contextix hopes to be one small step toward it.

---

## Final Principle

Software should not only preserve code.

It should preserve understanding.
